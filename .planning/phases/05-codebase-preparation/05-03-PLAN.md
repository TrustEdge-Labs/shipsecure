  <name>Task 1: Update backend Dockerfile with scanner binaries</name>
  <files>Dockerfile</files>
  <action>
Update the existing multi-stage Dockerfile to install Nuclei and testssl.sh in the runtime stage.

**Build stage** (keep existing, with minor improvements):
- Base: `rust:1.88-slim-bookworm AS builder` (keep)
- Install build deps: `pkg-config libssl-dev` (keep)
- Dependency caching layer (keep existing pattern)
- Full build (keep)

**Runtime stage** (update significantly):
- Base: `debian:bookworm-slim`
- Install runtime dependencies:
  ```
  apt-get update && apt-get install -y --no-install-recommends \
      ca-certificates \
      wget \
      unzip \
      git \
      bash \
      openssl \
      procps \
      curl \
  && rm -rf /var/lib/apt/lists/*
  ```

- Install Nuclei binary:
  ```
  RUN NUCLEI_VERSION=$(curl -s https://api.github.com/repos/projectdiscovery/nuclei/releases/latest | grep '"tag_name"' | sed 's/.*"v\(.*\)".*/\1/') \
      && wget -q "https://github.com/projectdiscovery/nuclei/releases/download/v${NUCLEI_VERSION}/nuclei_${NUCLEI_VERSION}_linux_amd64.zip" -O /tmp/nuclei.zip \
      && unzip /tmp/nuclei.zip -d /usr/local/bin/ nuclei \
      && chmod +x /usr/local/bin/nuclei \
      && rm /tmp/nuclei.zip
  ```

  If the dynamic version approach is fragile, pin a specific version. Use whatever is latest at time of execution. The pattern should be clear enough for future updates.

- Install testssl.sh:
  ```
  RUN git clone --depth 1 https://github.com/drwetter/testssl.sh.git /opt/testssl.sh \
      && ln -s /opt/testssl.sh/testssl.sh /usr/local/bin/testssl.sh \
      && chmod +x /opt/testssl.sh/testssl.sh
  ```

- Create non-root user:
  ```
  RUN useradd -m -u 1000 -U -s /bin/bash trustedge
  ```

- Copy artifacts from builder:
  ```
  COPY --from=builder /app/target/release/trustedge_audit /usr/local/bin/
  COPY --from=builder /app/migrations /app/migrations
  ```

- Copy templates directory (for vibe-code scanning):
  ```
  COPY templates /app/templates
  ```

- Copy fonts directory (for PDF generation):
  ```
  COPY fonts /app/fonts
  ```
  (If fonts/ doesn't exist yet, skip this line -- it's a pre-launch TODO)

- Set ownership, user, workdir, expose, cmd:
  ```
  RUN chown -R trustedge:trustedge /app
  USER trustedge
  WORKDIR /app
  EXPOSE 3000
  CMD ["trustedge_audit"]
  ```

- Set scanner environment defaults so they're discoverable inside the container:
  ```
  ENV NUCLEI_BINARY_PATH=/usr/local/bin/nuclei
  ENV TESTSSL_BINARY_PATH=/usr/local/bin/testssl.sh
  ENV TRUSTEDGE_TEMPLATES_DIR=/app/templates/nuclei
  ```

**Note:** Do NOT install Docker CLI in the runtime stage. Scanners now run as native binaries per Plan 01, not Docker containers.

   **frontend service** (keep existing pattern):
   - Build from `frontend/Dockerfile`
   - Environment: `BACKEND_URL=http://backend:3000`, `NEXT_PUBLIC_BACKEND_URL=http://localhost:3000`
   - Port mapping: `3001:3001`
   - depends_on backend

   **volumes**: pgdata (keep)

2. **Create `docker-compose.prod.yml`** (production overrides):

   This file is used with `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d`.

   **db service overrides:**
   - Remove port mapping (no external access in prod)
   - Override POSTGRES_PASSWORD to use env var: `${DB_PASSWORD}`
   - Override POSTGRES_DB to `trustedge_prod`

   **backend service overrides:**
   - Remove port mapping (proxied through Nginx, not directly exposed)
   - Add `restart: unless-stopped`
   - Override DATABASE_URL: `postgres://trustedge:${DB_PASSWORD}@db:5432/trustedge_prod`
   - Remove template volume mount (use baked-in templates from Docker image)
   - Add resource limits (deploy.resources.limits: cpus '2.0', memory '2G')
   - Add logging config: `logging: { driver: "json-file", options: { max-size: "10m", max-file: "3" } }`

   **frontend service overrides:**
   - Remove port mapping (proxied through Nginx)
   - Add `restart: unless-stopped`
   - Override `NEXT_PUBLIC_BACKEND_URL` to use production domain (via env var)
   - Add resource limits and logging

   Format:
   ```yaml
   services:
     db:
       ports: !override []
       environment:
         POSTGRES_PASSWORD: ${DB_PASSWORD}
         POSTGRES_DB: trustedge_prod
     backend:
       ports: !override []
       restart: unless-stopped
       environment:
         DATABASE_URL: postgres://trustedge:${DB_PASSWORD}@db:5432/trustedge_prod
       volumes: !override []
       deploy:
         resources:
           limits:
             cpus: '2.0'
             memory: 2G
       logging:
         driver: "json-file"
         options:
           max-size: "10m"
           max-file: "3"
     frontend:
       ports: !override []
       restart: unless-stopped
       environment:
         NEXT_PUBLIC_BACKEND_URL: ${PUBLIC_URL}
       deploy:
         resources:
           limits:
             cpus: '1.0'
             memory: 1G
       logging:
         driver: "json-file"
         options:
           max-size: "10m"
           max-file: "3"
   ```

   Note: The `!override` syntax may not work in all Compose versions. If needed, use empty arrays or just omit overrides that can't be cleared. Research the correct approach for the compose file format version being used. The key override pattern is: prod removes port mappings, adds restart policies, overrides env vars with production values.
