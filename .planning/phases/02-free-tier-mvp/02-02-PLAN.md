Create `src/scanners/tls.rs` implementing SCAN-03 (TLS/certificate analysis via SSL Labs API).

**Design:**
- Public async function: `pub async fn scan_tls(url: &str) -> Result<Vec<Finding>, ScannerError>`
- Extract hostname from the URL (use the `url` crate already in Cargo.toml)
- Use SSL Labs API v4: `https://api.ssllabs.com/api/v4/analyze?host={hostname}&startNew=on&all=done`
- Reuse `ScannerError` from `security_headers.rs` (move the error type to a shared location or duplicate it in tls.rs for now — keep it simple, share later if needed)

**SSL Labs API flow:**
1. Start assessment: GET `https://api.ssllabs.com/api/v4/analyze?host={hostname}&startNew=on`
2. Poll for completion: GET `https://api.ssllabs.com/api/v4/analyze?host={hostname}` every 10 seconds
3. Check response `status` field: "DNS" or "IN_PROGRESS" means keep polling, "READY" means done, "ERROR" means fail
4. Read rate limit headers from EVERY response: `X-Max-Assessments`, `X-Current-Assessments`
5. If `X-Current-Assessments >= X-Max-Assessments`, wait 30 seconds before retrying
6. Maximum poll attempts: 30 (5 minutes total at 10s intervals)
7. If status is "ERROR", return a single finding noting the TLS scan failed

**Finding generation from SSL Labs response (when status == "READY"):**
Parse the JSON response. For each endpoint in `endpoints[]`:
- Check `grade`: If grade is "F" or "T" (trust issues), create Critical finding. If "C" or "D", High. If "B", Medium. "A-" or below "A+" is Low. "A" or "A+" means no grade finding.
- Check `details.protocols[]`: If SSLv2 or SSLv3 enabled, Critical finding. If TLS 1.0 or 1.1 enabled, High finding.
- Check `details.cert.notAfter`: If certificate expires within 30 days, High finding. If expired, Critical.
- Check `details.heartbleed`: If true, Critical finding.
- Check `details.poodle`: If true, High finding.

Each finding should have:
- `scanner_name`: "tls"
- `severity`: as described above
- `description`: Plain-language explanation of the issue
- `remediation`: Actionable fix (e.g., "Disable TLS 1.0 and 1.1 in your server configuration. For Nginx: ssl_protocols TLSv1.2 TLSv1.3;")

**Rate limiting implementation:**
- Track `max_assessments` and `current_assessments` from response headers
- If approaching limit, add extra delay between polls
- On 429 response, wait 60 seconds and retry (max 3 retries for 429)
- On 529 (overloaded), wait 30 seconds

**Error handling:**
- DNS resolution failure: Return informational finding "Could not resolve hostname for TLS analysis"
- API timeout (5 min): Return informational finding "TLS analysis timed out"
- 429/529: Return informational finding "SSL Labs API rate limited, TLS results unavailable"

Add `base64` crate to Cargo.toml dependencies (needed for token generation in other plans, but add it now to avoid merge conflicts): `base64 = "0.22"`
Also add `rand = "0.8"` for token generation.

Update `src/scanners/mod.rs` to include: `pub mod tls;`

**Important:** The SSL Labs API is free and does NOT require an API key. Do not add any API key configuration. The API is rate-limited by IP address via the response headers mentioned above.

Create `src/scanners/exposed_files.rs` implementing SCAN-04 (exposed file/directory detection).

**Design:**
- Public async function: `pub async fn scan_exposed_files(url: &str) -> Result<Vec<Finding>, ScannerError>`
- Parse the base URL using the `url` crate (strip path, keep scheme + host)
- Probe a list of common sensitive paths using HTTP HEAD requests (fall back to GET if HEAD returns 405)

**Paths to probe (with expected findings):**

| Path | Finding Title | Severity | Description |
|------|--------------|----------|-------------|
| `/.env` | Exposed Environment File | Critical | ".env files often contain database credentials, API keys, and secrets. This file is publicly accessible." |
| `/.git/config` | Exposed Git Repository | Critical | "The .git directory is accessible, potentially exposing source code, commit history, and credentials." |
| `/.git/HEAD` | Exposed Git Repository | Critical | Same as above (check both, deduplicate) |
| `/wp-admin/` | Exposed Admin Panel | High | "An admin panel is publicly accessible without additional protection." |
| `/admin/` | Exposed Admin Panel | High | Same category |
| `/debug/` | Exposed Debug Endpoint | High | "Debug endpoints can expose internal application state, stack traces, and configuration." |
| `/_debug/` | Exposed Debug Endpoint | High | Same |
| `/api/debug` | Exposed Debug Endpoint | High | Same |
| `/.env.local` | Exposed Environment File | Critical | Same as .env |
| `/.env.production` | Exposed Environment File | Critical | Same |
| `/robots.txt` | Informational: robots.txt Found | Low | "robots.txt is accessible. Review it to ensure it doesn't reveal sensitive paths." (only if it contains Disallow directives pointing to interesting paths) |
| `/sitemap.xml` | Informational: sitemap.xml Found | Low | "sitemap.xml is accessible. Verify it doesn't expose internal or staging URLs." |
| `/.well-known/security.txt` | No security.txt Found | Low | Only create finding if NOT found (missing security contact) |
| `/server-status` | Exposed Server Status | High | "Apache server-status page is publicly accessible, revealing active connections and server configuration." |
| `/phpinfo.php` | Exposed PHP Info | High | "phpinfo() output is accessible, revealing server configuration, PHP version, and loaded modules." |
| `/*.js.map` | Exposed Source Maps | Medium | Check for common source map patterns: `/main.js.map`, `/app.js.map`, `/bundle.js.map` |

**Detection logic:**
- HTTP status 200 = file exists (create finding)
- HTTP status 403 = exists but blocked (no finding, good)
- HTTP status 404/410 = does not exist (no finding)
- HTTP status 301/302 to login page = no finding (properly protected)
- For `.env` files: Also check if response body contains common env patterns like `DB_`, `API_KEY`, `SECRET` to reduce false positives (a 200 with HTML is likely a framework catch-all, not a real .env)
- For `.git/config`: Check if response contains `[core]` or `[remote` to confirm it's actually a git config
- For `robots.txt`: Only create finding if response contains `Disallow:` with paths other than just `/`
- For source maps: Check Content-Type includes `application/json` or response starts with `{"version":3`

**Concurrency:** Run all path probes concurrently using `tokio::join!` or `futures::future::join_all`. Use a reqwest client with 10-second timeout per request.

**Error handling:**
- Connection errors for individual paths: Skip silently (don't create finding)
- All paths fail (host unreachable): Return error via ScannerError
- Timeout on individual paths: Skip that path

Each finding should have `scanner_name: "exposed_files"`, appropriate severity, plain-language description, and remediation like "Block access to this path in your web server configuration. For Nginx: location ~ /\\.env { deny all; return 404; }".
