<task type="auto">
  <name>Task 1: Configure Stripe test keys on production and validate checkout session creation</name>
  <files></files>
  <action>
**Pre-requisite: User must have Stripe test-mode keys ready.** The user_setup section documents what's needed. Before proceeding, confirm the user has:
- STRIPE_SECRET_KEY (starts with `sk_test_`)
- STRIPE_WEBHOOK_SECRET (starts with `whsec_`)
- Webhook endpoint created in Stripe Dashboard pointing to `https://shipsecure.ai/api/v1/webhooks/stripe` listening for `checkout.session.completed`

**Step 1: Configure Stripe keys on production**

SSH into production and add Stripe keys to the .env file:
```
ssh -p 2222 deploy@shipsecure.ai
```

Check current .env for Stripe configuration:
```
grep STRIPE /opt/trustedge/.env
```

If STRIPE_SECRET_KEY and STRIPE_WEBHOOK_SECRET are missing or empty, add them:
```
# Add to /opt/trustedge/.env (user provides actual values)
echo 'STRIPE_SECRET_KEY=sk_test_XXXXX' >> /opt/trustedge/.env
echo 'STRIPE_WEBHOOK_SECRET=whsec_XXXXX' >> /opt/trustedge/.env
```

Restart the backend to pick up new env vars:
```
sudo systemctl restart trustedge.service
```

Wait for containers to come up (10-15 seconds), then verify:
```
docker ps --format "{{.Names}} {{.Status}}" | grep trustedge
```

**Step 2: Verify Stripe configuration is loaded**

Check backend logs for any Stripe initialization errors:
```
docker logs trustedge-backend-1 --tail 50 2>&1 | grep -i stripe
```
No errors expected. The Stripe client is created on-demand per request, so no startup error is expected even if keys are wrong — errors will appear during checkout attempt.

**Step 3: Create a checkout session**

Use the scan_id from Plan 01's completed scan. The checkout endpoint requires a `scan_id` parameter which is actually the `results_token` from the scan.

First, get the results_token from the scan:
```
curl -sS https://shipsecure.ai/api/v1/scans/{SCAN_ID_FROM_PLAN_01} | jq '.results_token'
```

Then create a checkout session:
```
curl -sS -X POST https://shipsecure.ai/api/v1/checkout \
  -H "Content-Type: application/json" \
  -d '{"scan_id": "RESULTS_TOKEN_HERE"}'
```

Expected response: JSON with `checkout_url` field pointing to `https://checkout.stripe.com/...`

If this fails:
- "Stripe not configured": STRIPE_SECRET_KEY not loaded — check .env and restart
- "Scan not found": Wrong results_token — verify the token value
- "Scan must be completed": Scan status isn't "completed" — use a completed scan
- Stripe API error: Invalid API key — verify key starts with `sk_test_`

Record the checkout_url for the human verification step.

**Step 4: Verify paid_audit record created in database**

Check that the checkout session created a pending paid_audit record:
```
ssh -p 2222 deploy@shipsecure.ai 'docker exec trustedge-backend-1 env | grep DATABASE_URL'
```
Note the DATABASE_URL, then check the database (this might not be easily queryable from the container — alternatively check backend logs):
```
ssh -p 2222 deploy@shipsecure.ai 'docker logs trustedge-backend-1 --tail 20'
```
Look for log lines about "paid_audit" or "checkout session" creation.

  </verify>

<task type="auto">
  <name>Task 3: Validate service resilience — systemd restart and container crash recovery</name>
  <files></files>
  <action>
Per user decision: test service recovery by actually killing containers and verifying systemd restarts them.

**Scenario 1: Graceful systemd restart**

1. Check current service status:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'sudo systemctl status trustedge.service'
   ```
   Expected: "active (exited)" (oneshot with RemainAfterExit=yes)

2. Restart the service:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'sudo systemctl restart trustedge.service'
   ```

3. Wait 15 seconds for containers to come up:
   ```
   sleep 15
   ```

4. Verify service is active again:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'sudo systemctl is-active trustedge.service'
   ```
   Expected: "active"

5. Verify both containers are running:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'docker ps --format "{{.Names}} {{.Status}}" | grep trustedge'
   ```
   Expected: Both containers showing "Up"

6. Verify application responds:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'curl -s http://localhost:3000/health'
   ```
   Expected: "ok"

7. Verify external access:
   ```
   curl -sSf -o /dev/null -w "%{http_code}" https://shipsecure.ai
   ```
   Expected: 200

**Scenario 2: Container crash recovery**

Note: The systemd service uses Type=oneshot with RemainAfterExit=yes. Docker Compose restart policies were removed (systemd manages lifecycle). When a container is killed, Docker Compose itself may or may not restart it depending on the exit code. The key test is whether the application can be recovered via systemctl.

1. Kill the backend container directly:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'docker kill trustedge-backend-1'
   ```

2. Wait 10 seconds:
   ```
   sleep 10
   ```

3. Check if the container auto-recovered (Docker Compose may restart it):
   ```
   ssh -p 2222 deploy@shipsecure.ai 'docker ps --format "{{.Names}} {{.Status}}" | grep trustedge-backend'
   ```

4. If the container did NOT auto-recover, restart via systemctl:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'sudo systemctl restart trustedge.service'
   sleep 15
   ```

5. Verify recovery:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'docker ps --format "{{.Names}} {{.Status}}" | grep trustedge'
   ssh -p 2222 deploy@shipsecure.ai 'curl -s http://localhost:3000/health'
   curl -sSf -o /dev/null -w "%{http_code}" https://shipsecure.ai
   ```
   All must pass.

**Scenario 3: Full stop/start cycle**

1. Stop the service completely:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'sudo systemctl stop trustedge.service'
   ```

2. Verify containers are down:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'docker ps --format "{{.Names}}" | grep trustedge'
   ```
   Expected: no output (no running trustedge containers)

3. Start the service:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'sudo systemctl start trustedge.service'
   ```

4. Wait 15 seconds for startup:
   ```
   sleep 15
   ```

5. Verify full recovery:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'sudo systemctl is-active trustedge.service'
   ssh -p 2222 deploy@shipsecure.ai 'docker ps --format "{{.Names}} {{.Status}}" | grep trustedge'
   ssh -p 2222 deploy@shipsecure.ai 'curl -s http://localhost:3000/health'
   curl -sSf -o /dev/null -w "%{http_code}" https://shipsecure.ai
   ```
   All must pass: service active, both containers up, health check ok, external access 200.

6. Check journalctl for any errors during restart cycles:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'journalctl -u trustedge.service --since "10 minutes ago" --no-pager'
   ```
   Expected: clean start/stop/restart entries with no error lines.

If any scenario fails, debug using journalctl and docker logs, fix the issue, and re-run the scenario.

  </action>
  <verify>
- `systemctl restart trustedge.service` results in both containers running and app healthy
- After `docker kill trustedge-backend-1`, service can be recovered (either auto or via systemctl restart)
- Full stop/start cycle completes cleanly with both containers up and app responding
- No errors in journalctl for trustedge.service during restart cycles
- External access via `https://shipsecure.ai` returns 200 after each recovery
  </verify>
  <done>All three resilience scenarios pass: graceful restart, container crash recovery, and full stop/start cycle. Services self-heal and application is accessible after each test.</done>
