OSV.dev /v1/querybatch request format:
```json
POST https://api.osv.dev/v1/querybatch
{ "queries": [{ "package": { "name": "express", "ecosystem": "npm" }, "version": "4.18.2" }] }
```

OSV.dev /v1/querybatch response format (positionally aligned with queries):
```json
{ "results": [{ "vulns": [{ "id": "GHSA-rv95-896h-c2vc", "modified": "2024-01-15T00:00:00Z" }] }] }
```

OSV.dev /v1/vulns/{id} response format (hydration):
```json
{
  "id": "GHSA-...",
  "summary": "Description text",
  "severity": [{ "type": "CVSS_V3", "score": "CVSS:3.1/AV:N/AC:H/..." }],
  "database_specific": { "severity": "HIGH" }
}
```

**query_batch method:**
```rust
pub async fn query_batch(&self, deps: &[ParsedDep]) -> Result<Vec<DepVulnMatch>, SupplyChainError>
```

Where `DepVulnMatch` is:
```rust
pub struct DepVulnMatch {
    pub dep: ParsedDep,       // the dep that was queried
    pub vuln_ids: Vec<String>, // OSV IDs found (empty = clean)
}
```

Logic:
1. Filter to Registry deps only (caller should already do this, but defensive filter).
2. Chunk into groups of 1000 per D-06.
3. For each chunk, create an async future that calls `query_chunk()`.
4. Fire all chunk futures in parallel via `futures::future::join_all` per D-07.
5. If any chunk returned Err, return `Err(SupplyChainError::ChunkFailure(...))` per D-08.
6. Collect all successful results into a flat Vec<DepVulnMatch>.

**query_chunk method (private):**
- Build `OsvBatchRequest` with ecosystem "npm" for all deps in chunk.
- POST to `{base_url}/v1/querybatch`.
- If response status is 5xx: retry once per D-08. If second attempt also fails (5xx or error): return `Err(SupplyChainError::ChunkFailure(...))`.
- On success: deserialize `OsvBatchResponse`. The `results` array is positionally aligned with the input queries. Zip them to create `Vec<DepVulnMatch>`.
- Treat absent or empty `vulns` array as no vulnerabilities.

**hydrate_vulns method:**
```rust
pub async fn hydrate_vulns(&self, vuln_ids: &[String]) -> Result<Vec<OsvVulnDetail>, SupplyChainError>
```

- Deduplicate vuln_ids (same ID may appear for multiple deps).
- Fire parallel GET requests to `{base_url}/v1/vulns/{id}` via join_all.
- On any failure: return `Err(SupplyChainError::OsvQuery(...))`. No retry for hydration calls (they are individual, and failures are rare).
- Return Vec<OsvVulnDetail>.

**Orchestration flow:**
1. Call `lockfile_parser::parse(content)?` to get `Vec<ParsedDep>`.
2. Partition deps: `registry_deps` (source == Registry) and `non_registry_deps` (everything else).
3. Call `osv_client.query_batch(&registry_deps)?` to get `Vec<DepVulnMatch>`.
4. Collect all unique non-MAL vuln IDs that need hydration. MAL- prefixed IDs do NOT need hydration (categorized by ID alone per D-10).
5. Call `osv_client.hydrate_vulns(&ids_to_hydrate)?` to get `Vec<OsvVulnDetail>`. Build a HashMap<String, OsvVulnDetail> for lookup.
6. For each DepVulnMatch:
   - If vuln_ids is empty -> add dep name+version to `no_known_issues`.
   - For each vuln_id: call `categorize_finding()` to determine tier, create DepFinding.
7. Non-registry deps -> add to `unscanned` as-is.
8. Build and return `SupplyChainScanResult` with `scanned_at: chrono::Utc::now().naive_utc()`.

**categorize_finding function (pub for testing):**
```rust
pub fn categorize_finding(vuln_id: &str, detail: Option<&OsvVulnDetail>) -> DepTier
```

Logic per D-10 (order matters):
1. If vuln_id starts with "MAL-" -> return DepTier::Infected (detail may be None for MAL- entries; that's OK).
2. If detail has severity[] array with a CVSS_V3 entry: extract CVSS base score. The score field is a CVSS vector string like "CVSS:3.1/AV:N/AC:L/...". To extract the numeric base score:
   - **Option A (preferred if cvss crate available):** Parse with `cvss` crate to get base score float.
   - **Option B (if cvss crate not added):** Skip CVSS vector parsing entirely; fall through to database_specific.severity check. This is acceptable because GHSA entries (the majority for npm) reliably include database_specific.severity.
   - The executor should check if `cvss` crate is in Cargo.toml. If not, use Option B and document the tradeoff. Do NOT add a new crate without checking.
   - If base score >= 7.0 -> return DepTier::Vulnerable.
3. If detail has database_specific.severity: normalize to uppercase. If "CRITICAL" or "HIGH" -> return DepTier::Vulnerable.
4. Otherwise -> return DepTier::Advisory. (Any OSV match that isn't MAL- and isn't high-severity is Advisory.)

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46-04 | Spoofing | osv_client.rs | mitigate | OSV base URL is hardcoded to api.osv.dev; no user-controlled URL components; HTTPS enforced by reqwest default |
| T-46-05 | Denial of Service | osv_client.rs | mitigate | 10-second timeout per request per D-09; single retry cap per D-08; ChunkFailure aborts scan rather than hanging |
| T-46-06 | Tampering | osv_client categorize_finding | mitigate | Use serde with #[serde(default)] for all OSV response fields; never unwrap() on external API data; treat missing severity as Advisory (safe default) |
| T-46-07 | Information Disclosure | osv_client.rs | accept | Package names sent to OSV.dev are already public npm packages; no PII or proprietary data in queries |
| T-46-08 | Denial of Service | supply_chain.rs | mitigate | Non-registry deps skipped from OSV queries (smaller batch size); hydration is parallel with timeout; no unbounded loops |
