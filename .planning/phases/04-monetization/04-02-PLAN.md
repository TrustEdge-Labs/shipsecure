<objective>
Implement Stripe Checkout Session creation and webhook handler for processing paid audit payments.

Purpose: This is the payment backend. Users click "Upgrade" on the free results page, the frontend calls POST /api/v1/checkout which creates a Stripe Checkout Session and returns the checkout URL. After payment, Stripe sends a webhook to POST /api/v1/webhooks/stripe which records the payment, updates the scan tier to "paid", and spawns a paid scan.

Output: Two new API endpoints (checkout + webhook) wired into the Axum router.
</objective>

Create `src/api/checkout.rs` with a `create_checkout` handler:

1. Request body: `CreateCheckoutRequest { scan_id: Uuid }` (Deserialize)
2. Response body: `CreateCheckoutResponse { checkout_url: String }` (Serialize)

Handler logic:
- Extract `scan_id` from JSON body
- Load scan from database via `db::scans::get_scan` -- validate it exists and status == "completed"
- Check if a paid_audit already exists for this scan (prevent duplicate purchases) via `db::paid_audits::get_paid_audit_by_scan_id`
- If paid_audit exists and status is "completed", return error "Paid audit already purchased"
- Create Stripe client: `stripe::Client::new(std::env::var("STRIPE_SECRET_KEY")?)`
- Create CheckoutSession via `stripe::CheckoutSession::create(&client, params)`:
  - mode: Payment
  - line_items: single item, price_data with currency USD, unit_amount 4900 ($49.00), product_data name "TrustEdge Deep Security Audit"
  - customer_email: scan.email
  - success_url: `{FRONTEND_URL}/payment/success?session_id={CHECKOUT_SESSION_ID}` -- the `{CHECKOUT_SESSION_ID}` is Stripe's template variable
  - cancel_url: `{FRONTEND_URL}/results/{results_token}`
  - metadata: HashMap with "scan_id" => scan.id.to_string(), "email" => scan.email
- Create paid_audit record in database with status "pending" via `db::paid_audits::create_paid_audit`
- Return 200 with `{ checkout_url: session.url }`

Environment variables used: STRIPE_SECRET_KEY, FRONTEND_URL (default "http://localhost:3001").

Error handling: Use existing ApiError enum. Add new variants as needed (StripeError, PaymentError) or use existing InternalError with descriptive messages. Return RFC 7807 errors consistent with existing patterns.

<task type="auto">
  <name>Task 2: Stripe webhook handler with signature verification and idempotency</name>
  <files>src/api/webhooks.rs, src/api/mod.rs, src/main.rs</files>
  <action>
1. Create `src/api/webhooks.rs` with `handle_stripe_webhook` handler:

Signature verification (manual, using hmac/sha2/hex -- more reliable than async-stripe's built-in):
- Extract `stripe-signature` header from HeaderMap
- Parse header to get `t` (timestamp) and `v1` (signature) values
- Compute expected signature: HMAC-SHA256(STRIPE_WEBHOOK_SECRET, "{timestamp}.{body}")
- Compare computed vs received signature using constant-time comparison
- Reject if timestamp is older than 5 minutes (replay protection)

Event processing:
- Parse body as `serde_json::Value`
- Extract event type from `data.type` field
- Extract event ID from `data.id` field
- Call `db::paid_audits::check_and_mark_event(pool, event_id)` -- if returns false (duplicate), return 200 OK immediately
- If event type is `checkout.session.completed`:
  - Extract checkout session from `data.object`
  - Extract `metadata.scan_id` and `metadata.email` from session object
  - Extract `payment_intent` from session object
  - Call `db::paid_audits::update_paid_audit_status(pool, checkout_session_id, "completed", payment_intent_id)`
  - Call `db::paid_audits::update_scan_tier(pool, scan_id, "paid")`
  - Spawn paid scan: `tokio::spawn(async move { ... })` -- this will be wired to the orchestrator's paid scan method in Plan 03. For now, just log "Paid scan triggered for {scan_id}" as a placeholder. The actual orchestrator integration happens in Plan 03.
- Return 200 OK immediately (Stripe best practice: acknowledge quickly)

IMPORTANT: The webhook handler must accept raw body bytes (not parsed JSON) for signature verification. Use `axum::body::Bytes` as the extractor. The raw body string is needed for HMAC computation before any JSON parsing.
