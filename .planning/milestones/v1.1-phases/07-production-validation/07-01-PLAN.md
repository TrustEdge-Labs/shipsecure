   ```
   Expected: 200. If not, debug Nginx/Certbot configuration via SSH (`ssh -p 2222 deploy@shipsecure.ai`).

2. Verify SSL certificate details (valid, not expired, correct domain):
   ```
   echo | openssl s_client -connect shipsecure.ai:443 -servername shipsecure.ai 2>/dev/null | openssl x509 -noout -dates -subject
   ```

3. Verify backend health endpoint (via SSH since /health is not exposed externally):
   ```
   ssh -p 2222 deploy@shipsecure.ai 'curl -s http://localhost:3000/health'
   ```
   Expected: "ok"

4. Verify both containers are running:

   ```
   Expected: Both trustedge-backend and trustedge-frontend containers show "Up"

**Stage 3: Scanner Validation via Full Scan**

Per user decision: test each of the 5 scanners individually by triggering a scan against a known-vulnerable target and inspecting per-scanner findings.

1. Submit a scan against testphp.vulnweb.com (known Acunetix test site, legal to scan, has missing security headers, exposed files, weak TLS):
   ```
   curl -sS -X POST https://shipsecure.ai/api/v1/scans \
     -H "Content-Type: application/json" \
     -d '{"url": "http://testphp.vulnweb.com", "email": "USER_EMAIL_HERE"}'
   ```
   **IMPORTANT:** Use the user's actual email address so they can verify email delivery in the next task. Ask the user for their email before submitting.

   Record the scan ID from the response.

2. Poll for scan completion (scans can take 1-5 minutes):
   ```
   curl -sS https://shipsecure.ai/api/v1/scans/{SCAN_ID} | jq '.status'
   ```
   Poll every 10 seconds until status is "completed". If status becomes "failed", check backend logs:
   ```
   ssh -p 2222 deploy@shipsecure.ai 'docker logs trustedge-backend-1 --tail 200'
   ```
   Debug and fix any scanner failures. Common issues:
   - Nuclei binary not found: verify NUCLEI_BINARY_PATH in .env
   - SSRF validation blocking testphp.vulnweb.com: check ssrf validation logic
   - Timeout: scan took too long, check Nginx proxy_read_timeout (should be 300s)

3. Once completed, inspect findings by scanner type:
   ```
   curl -sS https://shipsecure.ai/api/v1/scans/{SCAN_ID} | jq '.findings | group_by(.scanner_name) | .[] | {scanner: .[0].scanner_name, count: length}'
   ```
   All 5 scanners must return at least some findings:
   - security_headers: Missing CSP, X-Frame-Options, etc.
   - tls: TLS configuration issues (if any — some test targets have modern TLS, which is acceptable for this scanner to report fewer findings)
   - js_secrets: Potential hardcoded strings in JavaScript
   - exposed_files: Common exposed files (.git, .env, etc.)
   - vibecode: Framework detection (PHP for testphp.vulnweb.com)

   If any scanner returns 0 findings, investigate via backend logs. Zero findings for TLS scanner is acceptable if the target has good TLS — the scanner working correctly with 0 findings is different from the scanner failing silently. Check logs to distinguish.

4. If any scanner is broken, debug and fix:
   - Check backend logs for that scanner's output
   - Verify scanner binary exists and is accessible
   - Fix the issue in code if needed, rebuild, redeploy
   - Re-run the scan to confirm the fix
