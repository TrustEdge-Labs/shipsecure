  <name>Task 1: Nginx CVE-2025-29927 mitigation — strip x-middleware-subrequest header</name>
  <files>
    infrastructure/templates/shipsecure.nginx.conf.j2
  </files>
  <action>
Add `proxy_set_header x-middleware-subrequest "";` to BOTH the `/api/` and `/` location blocks in the HTTPS server block of `infrastructure/templates/shipsecure.nginx.conf.j2`.

**CRITICAL:** The header strip MUST appear in BOTH location blocks. The CVE affects Next.js directly (served via the `/` block), but the `/api/` block also passes through Next.js middleware when requests arrive at the frontend proxy. Missing either block leaves the vulnerability exploitable.

In the `/api/` location block (after the existing `proxy_set_header` directives, before the timeout directives):
```nginx
    # CVE-2025-29927 mitigation — strip spoofed middleware bypass header
    proxy_set_header x-middleware-subrequest "";
```

In the `/` location block (after the existing `proxy_set_header` directives, before the timeout directives):
```nginx
    # CVE-2025-29927 mitigation — strip spoofed middleware bypass header
    proxy_set_header x-middleware-subrequest "";
```

Setting the header to an empty string causes Nginx to strip it before forwarding to upstream. This prevents external attackers from sending `x-middleware-subrequest: 1` to bypass Next.js proxy.ts auth middleware.

Keep ALL existing directives in both blocks unchanged. Only add the new line.
  </action>

Verify Nginx config syntax is valid by visual inspection (the template is Jinja2 so we cannot run nginx -t directly, but the additions are standard directives).
  </verify>
  <done>
