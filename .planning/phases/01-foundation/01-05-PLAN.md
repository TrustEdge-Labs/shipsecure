```yaml
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: trustedge

      interval: 5s
      timeout: 5s
      retries: 5

  pgdata:
```

**Dockerfile (multi-stage build):**
```dockerfile
# Build stage
FROM rust:1.82-slim-bookworm AS builder
WORKDIR /app
RUN apt-get update && apt-get install -y pkg-config libssl-dev && rm -rf /var/lib/apt/lists/*

# Cache dependencies
COPY Cargo.toml Cargo.lock ./
RUN mkdir src && echo "fn main() {}" > src/main.rs && echo "" > src/lib.rs
RUN cargo build --release 2>/dev/null || true
RUN rm -rf src

# Build application
COPY . .
ENV SQLX_OFFLINE=true
RUN cargo build --release

# Runtime stage
FROM debian:bookworm-slim
RUN apt-get update && apt-get install -y ca-certificates && rm -rf /var/lib/apt/lists/*
COPY --from=builder /app/target/release/trustedge-audit /usr/local/bin/
COPY --from=builder /app/migrations /app/migrations
WORKDIR /app
EXPOSE 3000
CMD ["trustedge-audit"]
```

Note on SQLX_OFFLINE: The Dockerfile uses SQLX_OFFLINE=true which requires a `.sqlx/` directory with cached query metadata. If Plan 04 didn't generate this, either:
1. Run `cargo sqlx prepare` locally before Docker build (preferred)
2. Or switch to runtime-checked queries (`sqlx::query` instead of `sqlx::query!` macro)

**Update .env.example** to include:
```
DATABASE_URL=postgres://trustedge:trustedge@localhost:5432/trustedge_dev
PORT=3000
RUST_LOG=info,trustedge_audit=debug
```

After creating Docker files, start the database and run the full application:

2. Wait for health check to pass

This task is where integration issues surface. Common problems to watch for:

- Axum extractor ordering issues (State must come before Json in handler signatures)
- Missing trait implementations (Clone, Send, Sync for types shared across async boundaries)

Fix all issues until `cargo run` starts successfully and the health endpoint responds.

<task type="auto">
  <name>Task 2: End-to-end verification of all Phase 1 success criteria</name>

  <action>
With the application running (from Task 1), verify all 5 Phase 1 success criteria:

**SC1: Developer can POST a URL to /api/v1/scans and receive a scan ID**
```bash
curl -s -X POST http://localhost:3000/api/v1/scans \
  -H "Content-Type: application/json" \
  -d '{"url":"https://httpbin.org","email":"test@example.com"}' | jq .
```
Expected: 201 response with `id`, `status: "pending"`, and `url` fields.
Save the returned scan ID for subsequent tests.

**SC2: Backend executes security headers scan and stores findings in PostgreSQL**
Wait 5-10 seconds for the scan to complete, then:
```bash
curl -s http://localhost:3000/api/v1/scans/{SCAN_ID} | jq .
```
Expected: status is "completed", findings array is non-empty, each finding has title, severity, remediation.
Verify findings are in the database: `docker compose exec db psql -U trustedge -d trustedge_dev -c "SELECT title, severity FROM findings WHERE scan_id = '{SCAN_ID}'"`

**SC3: Developer can GET /api/v1/scans/:id and retrieve scan status and results**
Verify the GET response includes:
- scan metadata (id, target_url, status, score, timestamps)
- findings array with all fields
- summary object with counts by severity
Test 404: `curl -s http://localhost:3000/api/v1/scans/00000000-0000-0000-0000-000000000000` -- should return 404 with RFC 7807 body.

**SC4: System blocks scanning of localhost, internal IPs, and cloud metadata endpoints**
Test each blocked target:
```bash
# Localhost
curl -s -X POST http://localhost:3000/api/v1/scans \
  -H "Content-Type: application/json" \
  -d '{"url":"http://127.0.0.1","email":"test@example.com"}' | jq .

# Private IP
curl -s -X POST http://localhost:3000/api/v1/scans \
  -H "Content-Type: application/json" \
  -d '{"url":"http://192.168.1.1","email":"test@example.com"}' | jq .

# Cloud metadata
curl -s -X POST http://localhost:3000/api/v1/scans \
  -H "Content-Type: application/json" \
  -d '{"url":"http://169.254.169.254/latest/meta-data/","email":"test@example.com"}' | jq .
```
Expected: All return 400 with RFC 7807 error body containing "ssrf-blocked" type.

**SC5: System enforces rate limits (max 3 scans per email per day)**
Submit 4 scans with the same email (use a different email than SC1 to avoid prior counts):
```bash
for i in 1 2 3 4; do
  echo "--- Scan $i ---"
  curl -s -X POST http://localhost:3000/api/v1/scans \
    -H "Content-Type: application/json" \
    -d '{"url":"https://example.com","email":"ratelimit@test.com"}' | jq .status
done
```
Expected: First 3 return 201 (or the response code for success), 4th returns 429 with "rate-limited" error.

If any test fails, diagnose and fix the issue in the relevant source file. Common fixes:
- Timing issue: scan not completed yet -> increase wait time or poll
- SSRF not blocking: check that DNS resolution is actually happening for IP literals
- Rate limit not counting: verify the count query uses CURRENT_DATE correctly
- Missing findings: check orchestrator is calling scanner and persisting results

Document all test results.
