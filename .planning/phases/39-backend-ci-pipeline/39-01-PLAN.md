<task type="auto">
  <name>Task 1: Add backend-ci job with test, clippy, and fmt checks</name>
  <files>.github/workflows/ci.yml</files>
  <action>
Add a new job `backend-ci` to `.github/workflows/ci.yml` that runs on every push/PR to main (same triggers already in the workflow). The job must:

1. **Use `ubuntu-latest` runner** with no `working-directory` default (Cargo.toml is at repo root).

2. **Install Rust stable toolchain** using `dtolnay/rust-toolchain@stable` with components: `clippy, rustfmt`. This action is the standard for GitHub Actions Rust CI (faster than rustup manually, handles caching of toolchain).

3. **Cache Cargo dependencies** using `actions/cache@v5` with:
   - `path: |`
     - `~/.cargo/bin/`
     - `~/.cargo/registry/index/`
     - `~/.cargo/registry/cache/`
     - `~/.cargo/git/db/`
     - `target/`
   - `key: ${{ runner.os }}-cargo-${{ hashFiles('Cargo.lock') }}`

4. **Run cargo fmt check**: `cargo fmt --all -- --check`
   - This must be the FIRST check (fastest, catches formatting issues immediately).
   - Fails if any file is not formatted.

5. **Run cargo clippy**: `cargo clippy --all-targets --all-features -- -D warnings`
   - `-D warnings` promotes all warnings to errors, failing the build.
   - `--all-targets` includes test code in linting.

6. **Run cargo test**: `cargo test --all-targets`
   - Runs all tests except ignored ones.
   - The backend tests in ssrf/validator.rs, scanners/, api/errors.rs etc. do NOT require a database (they test pure logic).
   - The rate_limit/middleware.rs tests are already `#[ignore]` (require PostgreSQL), so they will be skipped automatically.

Place the `backend-ci` job BEFORE the existing `unit-tests` job in the file. Do NOT modify the existing `unit-tests` or `e2e-tests` jobs. The backend-ci job runs independently (no `needs:` dependency on frontend jobs and vice versa).

Do NOT add `--all-features` to cargo test (sqlx features are already specified in Cargo.toml).
  </action>
  <verify>
    <automated>grep -q "backend-ci" .github/workflows/ci.yml && grep -q "cargo fmt" .github/workflows/ci.yml && grep -q "cargo clippy" .github/workflows/ci.yml && grep -q "cargo test" .github/workflows/ci.yml && echo "PASS" || echo "FAIL"</automated>
  </verify>
  <done>The ci.yml file contains a backend-ci job with cargo fmt --check, cargo clippy -D warnings, and cargo test steps. The existing frontend jobs are unchanged.</done>
</task>

<task type="auto">
  <name>Task 2: Add backend coverage reporting with cargo-llvm-cov</name>
  <files>.github/workflows/ci.yml</files>
  <action>
Add a second backend job `backend-coverage` to `.github/workflows/ci.yml` that generates a coverage report visible in CI output. This job runs AFTER `backend-ci` passes (add `needs: [backend-ci]`).

1. **Use `ubuntu-latest` runner**.

2. **Install Rust stable toolchain** using `dtolnay/rust-toolchain@stable` with component: `llvm-tools-preview` (required by cargo-llvm-cov).

3. **Install cargo-llvm-cov**: Use `taiki-e/install-action@cargo-llvm-cov` (official installer action, avoids slow compilation from source).

4. **Cache Cargo dependencies** (same cache config as backend-ci job — the cache key matches so it will restore from the prior job's cache).

5. **Run coverage**: `cargo llvm-cov --all-targets --ignore-filename-regex "main\.rs" -- --skip ignored`
   - `--ignore-filename-regex "main\.rs"` excludes the binary entrypoint from coverage (it just calls lib functions).
   - `-- --skip ignored` passes through to cargo test to skip `#[ignore]` tests.
   - This prints a text summary to stdout, which is visible in the GitHub Actions log.

6. **IMPORTANT: Do NOT set coverage thresholds** — the REQUIREMENTS.md explicitly marks "Backend coverage thresholds" as out of scope. This is report-only. Do NOT use `--fail-under-lines` or similar flags.

7. The job should NOT block the build on low coverage — it only reports. If cargo-llvm-cov itself fails (e.g., compilation error), the job should fail.

The coverage report output will be visible directly in the GitHub Actions job log, satisfying CI-04's requirement that coverage is "visible in CI output."
  </action>
  <verify>
    <automated>grep -q "backend-coverage" .github/workflows/ci.yml && grep -q "cargo-llvm-cov\|llvm-cov" .github/workflows/ci.yml && grep -q "needs:.*backend-ci\|needs: \[backend-ci\]" .github/workflows/ci.yml && echo "PASS" || echo "FAIL"</automated>
  </verify>
  <done>The ci.yml file contains a backend-coverage job that depends on backend-ci, installs cargo-llvm-cov, runs coverage, and prints results to CI output. No coverage threshold enforcement.</done>
</task>
