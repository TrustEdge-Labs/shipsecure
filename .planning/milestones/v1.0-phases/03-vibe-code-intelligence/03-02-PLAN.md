  <action>
Create the `templates/nuclei/` directory and write 7 custom Nuclei v3 YAML templates. All templates must use `tags: vibe-code` for identification.

**1. `supabase-rls.yaml` - Supabase RLS Misconfiguration:**
- First request: GET {{BaseURL}}, extract Supabase project URL and anon key from HTML/JS
  - Match regex for Supabase URL: `https://[a-z0-9-]+\.supabase\.co`
  - Match regex for anon key (JWT): `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9\.[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+`
  - Use extractors with `internal: true` to capture values
- Second request: GET `https://{{supabase_url}}/rest/v1/users?select=*&limit=1` with apikey and Authorization headers using extracted anon key
  - Matchers: status 200 AND response contains field names suggesting private data (email, password, user_id, auth)
  - Severity: critical
  - PITFALL: Do NOT flag if response is empty array or contains only public-looking fields
  - Tags: `vibe-code,supabase,rls,baas`
  - Info description must reference CVE-2025-48757 class

**2. `firebase-rules.yaml` - Firebase Permissive Rules:**
- Extract Firebase project from HTML (firebaseConfig, firebaseapp.com patterns)
- Probe `https://{{project}}.firebaseio.com/.json` (GET)
- If returns 200 with JSON data (not "Permission denied"), flag as permissive rules
- Severity: critical for full access, high if partial
- Tags: `vibe-code,firebase,baas`

**3. `nextjs-env-leak.yaml` - NEXT_PUBLIC_ Secret Exposure:**
- GET {{BaseURL}} and scan response body for NEXT_PUBLIC_ variables containing secret-like values
- Match patterns: NEXT_PUBLIC_.*SECRET, NEXT_PUBLIC_.*PRIVATE, NEXT_PUBLIC_.*SERVICE_ROLE, NEXT_PUBLIC_.*API_KEY (but NOT NEXT_PUBLIC_SUPABASE_URL or NEXT_PUBLIC_SUPABASE_ANON_KEY -- these are safe)
- Use negative matchers to whitelist safe patterns per research pitfall #4
- Severity: high (secrets in client bundle)
- Tags: `vibe-code,nextjs,env-leak`

**4. `unprotected-api-routes.yaml` - Unprotected API Routes:**
- Probe common Next.js API paths: /api/users, /api/admin, /api/auth/user, /api/settings
- Also probe App Router patterns: /api/user/route, /api/admin/route
- Match: 200 status with JSON response body (not 401/403)
- Severity: high (API routes returning data without auth)
- Tags: `vibe-code,api,auth`
- PITFALL: Check BOTH /pages/api/* AND /app/*/route patterns per research

**5. `env-in-build-output.yaml` - .env in Build Output:**
- Probe: {{BaseURL}}/.env, {{BaseURL}}/.env.local, {{BaseURL}}/.env.production
- Match: 200 status with content containing env-like patterns (KEY=value lines)
- Severity: critical (.env files should never be served)
- Tags: `vibe-code,env-leak,misconfiguration`
- Note: This overlaps with exposed_files scanner but specifically targets .env patterns common in vibe-coded deploys

**6. `netlify-function-exposure.yaml` - Netlify Function Exposure:**
- Probe: {{BaseURL}}/.netlify/functions/, {{BaseURL}}/.netlify/functions-internal/
- Match: 200 status revealing function list or debug info
- Severity: medium
- Tags: `vibe-code,netlify,functions`

**7. `vercel-env-leak.yaml` - Vercel Environment Leak:**
- Scan page source for VERCEL_ prefixed vars in client bundle (VERCEL_ENV, VERCEL_URL)
- Also check for NEXT_PUBLIC_VERCEL_ patterns that leak deployment info
- Severity: low (info disclosure, not secrets per se)
- Tags: `vibe-code,vercel,env-leak`

All templates MUST:
- Use Nuclei v3 syntax (matchers-condition: and/or)
- Include `info.tags: vibe-code` for identification
- Include `info.author: trustedge-audit`
- Include `info.reference` with relevant CVE or documentation links
- Use only passive + light active probing (no brute force, no heavy fuzzing)

  <action>
1. Add `pub mod vibecode;` to `src/scanners/mod.rs` (alongside existing modules).

2. Create `src/scanners/vibecode.rs` implementing the vibe-code vulnerability scanner:

   a. Define `pub async fn scan_vibecode(target_url: &str, framework: Option<&str>, platform: Option<&str>) -> Result<Vec<Finding>, ScannerError>`:
      - This function runs Nuclei with custom vibe-code templates against the target
      - The `framework` and `platform` params are optional strings (e.g., "nextjs", "vercel") from detection results
      - These are used to select which templates to run

   b. Template selection logic:
      - ALWAYS run: `supabase-rls.yaml`, `firebase-rules.yaml`, `env-in-build-output.yaml` (universal vibe-code checks)
      - If framework == "nextjs": also run `nextjs-env-leak.yaml`, `unprotected-api-routes.yaml`
      - If platform == "vercel": also run `vercel-env-leak.yaml`
      - If platform == "netlify": also run `netlify-function-exposure.yaml`
      - If framework is None: run ALL templates (can't narrow, so be comprehensive)

   c. Docker execution:
      - Use the existing `container.rs` pattern for Docker execution with CIS security hardening flags
      - Mount templates directory as read-only volume: `-v {templates_dir}:/templates:ro`
      - Get templates_dir from current working directory or env var `TRUSTEDGE_TEMPLATES_DIR`
      - Use `projectdiscovery/nuclei:latest` image
      - Pass `-t /templates/` to use custom templates only
      - Pass `-jsonl` for JSON line output
      - Pass `-severity critical,high,medium,low`
      - Pass `-timeout 30` (per-request timeout)
      - Container timeout: 120 seconds
      - If Docker is not available, log warning and return empty vec (graceful degradation per project convention)

   d. Parse JSONL output into Finding structs:
      - Reuse/adapt the `parse_nuclei_output` and `parse_nuclei_finding` functions from container.rs
      - BUT set `vibe_code: true` on ALL findings from this scanner (this is the key differentiator)
      - Set `scanner_name: "vibecode"` (distinct from generic "nuclei" scanner in container.rs)
      - Map Nuclei severity to existing Severity enum

   e. Env variable leak whitelist (per CONTEXT.md and research):
      - After parsing findings, filter out false positives for known safe NEXT_PUBLIC_ vars:
        - NEXT_PUBLIC_SUPABASE_URL (publishable URL)
        - NEXT_PUBLIC_SUPABASE_ANON_KEY (publishable anon key)
      - Log filtered findings at debug level

   f. Add unit tests:
      - Test template selection with framework="nextjs" includes nextjs-specific templates
      - Test template selection with framework=None includes all templates
      - Test finding parsing from mock JSONL with vibe_code=true
      - Test whitelist filtering removes safe NEXT_PUBLIC_ vars

IMPORTANT: This scanner does NOT depend on Plan 01's detector module at runtime -- it takes framework/platform as simple string parameters. The orchestrator (Plan 04) will wire them together.
