<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Configure secrets and prerequisites</name>
  <action>
The following must be completed before running the Ansible playbook:

**1. DigitalOcean API token:**
   - Go to DigitalOcean Dashboard -> API -> Tokens -> Generate New Token
   - Token needs read+write permissions

**2. DigitalOcean Managed PostgreSQL:**
   - Go to DigitalOcean Dashboard -> Databases -> Create Database Cluster
   - Engine: PostgreSQL 16
   - Region: NYC3 (same as droplet)
   - Plan: Basic ($15/mo for MVP)
   - Create database `trustedge_prod` and user `trustedge` in Users & Databases tab
   - Copy the connection string (includes sslmode=require)

**3. SSH key:**
   - Ensure you have an SSH key pair locally: `ls ~/.ssh/id_ed25519.pub`
   - If not, generate one: `ssh-keygen -t ed25519`
   - Upload public key to DigitalOcean: Dashboard -> Settings -> Security -> Add SSH Key

**4. Domain DNS:**
   - Note: The droplet IP will be displayed during provisioning
   - After provisioning, create DNS A record: `yourdomain.com -> <droplet-ip>`
   - Wait for DNS propagation (5-15 minutes)

**5. Encrypt Ansible Vault:**
   ```bash
   cd infrastructure
   # Edit vault.yml with real values
   nano inventory/group_vars/vault.yml
   # Then encrypt it
   ansible-vault encrypt inventory/group_vars/vault.yml
   ```
  </action>
  <resume-signal>Type "ready" when secrets are configured, managed database is created, and vault is encrypted</resume-signal>
</task>

<task type="auto">
  <name>Task 2: Install Ansible dependencies and run playbook</name>
  <files></files>
  <action>
**Install Ansible requirements:**
```bash
cd infrastructure
pip3 install --user pydo azure-core boto3
ansible-galaxy collection install -r requirements.yml
```

**Run the provisioning playbook:**
```bash
ansible-playbook playbooks/provision.yml --ask-vault-pass
```

Monitor the output for:
- Droplet creation (displays IP address — user needs this for DNS)
- SSH hardening completion
- Docker installation
- Nginx installation and initial config
- Certbot SSL certificate (will fail if DNS not pointed yet — this is expected)

If Certbot fails because DNS is not yet pointed:
1. Note the droplet IP from the output
2. Wait for the user to point DNS
3. Re-run the playbook (it's idempotent — will skip completed tasks and retry Certbot)

**Verify infrastructure after successful run:**
```bash
# Test SSH access on non-standard port
ssh -p 2222 deploy@<droplet-ip>

# On the droplet, verify:
sudo ufw status                           # Should show 2222, 80, 443 allowed
docker --version                          # Docker installed
docker compose version                    # Compose v2 installed
systemctl status trustedge                # Service running
curl -s http://127.0.0.1:3000/health      # Backend responding
curl -s http://127.0.0.1:3001             # Frontend responding
curl -sI https://<domain>                 # HTTPS working
```
  </action>

All verification commands above return expected results:
- UFW shows only ports 2222, 80, 443 allowed
- Docker and Compose v2 installed
- trustedge.service active and running
- Backend health endpoint returns 200
- Frontend returns HTML
