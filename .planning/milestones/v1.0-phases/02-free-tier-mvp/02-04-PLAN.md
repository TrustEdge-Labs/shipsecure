Update `src/orchestrator/worker_pool.rs` to run all 4 scanners and track per-stage completion.

**Changes to `run_scanners` method:**
Replace the current implementation that only runs `security_headers` with:

1. Run all 4 scanners concurrently using `tokio::join!`:
   - `security_headers::scan_security_headers(target_url)` with 60s timeout
   - `tls::scan_tls(target_url)` with 300s timeout (SSL Labs can be slow)
   - `exposed_files::scan_exposed_files(target_url)` with 60s timeout
   - `js_secrets::scan_js_secrets(target_url)` with 60s timeout

2. After each scanner completes (success OR failure), call `db::scans::update_scan_stage(pool, scan_id, stage_name, true)` to mark that stage as done. Stage names: "headers", "tls", "files", "secrets".

3. Since scanners run concurrently via `tokio::join!`, all stages complete at roughly the same time. However, the stage updates still provide meaningful progress tracking because `tokio::join!` returns as each future completes internally. To get true per-stage updates, change from `tokio::join!` to spawning each scanner as a separate `tokio::spawn` task that updates its stage upon completion, then `await` all handles.

**Revised approach for per-stage tracking:**
```rust
// Spawn each scanner independently so stage updates happen as each completes
let headers_handle = {
    let pool = pool.clone();
    let url = target_url.to_string();
    tokio::spawn(async move {
        let result = tokio::time::timeout(Duration::from_secs(60),
            security_headers::scan_security_headers(&url)).await;
        let _ = scans_db::update_scan_stage(&pool, scan_id, "headers", true).await;
        // convert result to ScannerResult
        result
    })
};
// ... same pattern for tls, exposed_files, js_secrets
```

4. Collect findings from all successful scanners (same as current pattern)
5. The `run_scanner_with_retry` function signature needs to be generalized. Currently it's tied to `security_headers::ScannerError`. Make it generic over any error type that implements `Display`, OR create a shared `ScannerError` type. The simplest approach: keep the retry logic inline for each scanner rather than trying to make the generic function work across different scanner error types.

**Changes to `execute_scan_internal`:**
After computing score and persisting findings:
1. Generate a secure results token: Use `rand::thread_rng().gen::<[u8; 32]>()` then `base64::engine::general_purpose::URL_SAFE_NO_PAD.encode(&bytes)` to get a 43-char base64url token
2. Compute expiry: `chrono::Utc::now().naive_utc() + chrono::Duration::days(3)`
3. Call `db::scans::set_results_token(pool, scan_id, &token, expires_at)`
4. Send email notification (call the email module from Task 2)
5. The email needs: recipient email (from scan record), target URL, grade, finding counts, results token
6. Query the scan record to get the email address: `db::scans::get_scan(pool, scan_id)`
7. If email sending fails, log the error but do NOT fail the scan (findings are still available via results page)

**Update imports** in `worker_pool.rs` to include the new scanner modules: `use crate::scanners::{security_headers, tls, exposed_files, js_secrets};`

Make sure the pool is passed through to `execute_scan_internal` properly (it already is via `pool: PgPool` parameter). Add `scan_id: Uuid` parameter to `run_scanners` so stage updates can happen inside.

Update `src/orchestrator/mod.rs` to re-export what's needed.

**Resend API integration:**
- Endpoint: POST `https://api.resend.com/emails`
- Authorization: Bearer token from `RESEND_API_KEY` environment variable
- Request body (JSON):
  ```json
  {
    "from": "TrustEdge Audit <scans@trustedgeaudit.com>",
    "to": ["{email}"],
    "subject": "Scan Complete: {grade} Grade for {url}",
    "html": "{html_body}"
  }
  ```
- If `RESEND_API_KEY` is not set, log a warning and skip email sending (development mode)
- On API error, log the error and return EmailError (caller decides whether to propagate)
