**Step 2: Configure branch protection via `gh api`**

Run the following command to set branch protection on `main`:

```bash
gh api repos/{owner}/{repo}/branches/main/protection \
  --method PUT \
  --header "Accept: application/vnd.github+json" \
  -f "required_status_checks[strict]=false" \
  -f "required_status_checks[checks][][context]=Unit Tests" \
  -f "required_status_checks[checks][][context]=E2E Tests" \
  -F "enforce_admins=true" \
  -F "required_pull_request_reviews=null" \
  -F "restrictions=null"
```

Determine the correct owner/repo from the git remote (likely `TrustEdge-Labs/shipsecure`):
```bash
git remote get-url origin
```

Key details (all per locked decisions):
- `strict: false` — branch does not need to be up-to-date before merging (avoids forcing rebases)
- `checks` context must EXACTLY match job `name:` fields in ci.yml: `Unit Tests` and `E2E Tests` (not workflow name "CI")
- `enforce_admins: true` — no bypass, even for repo owner (strict enforcement per user decision)
- `required_pull_request_reviews: null` — no review required (sole developer per user decision)
- `restrictions: null` — no push restrictions

**Step 3: Verify branch protection is active**

```bash
gh api repos/{owner}/{repo}/branches/main/protection \
  --header "Accept: application/vnd.github+json" | jq '.required_status_checks'
```

Confirm:
- `checks` array contains `Unit Tests` and `E2E Tests`
- `enforce_admins.enabled` is `true`

**IMPORTANT:** If the CI workflow has not yet run on GitHub, the `gh api` call will succeed but the checks may not match existing observed checks. In that case, the protection will still be set but may show "never been run" in the GitHub UI. Once CI runs, it will match. This is expected behavior per research findings.
