    **TypeScript types** (`frontend/lib/types.ts`) — add these interfaces:

    ```typescript
    export interface VerifiedDomain {
      id: string
      domain: string
      status: 'pending' | 'verified'
      verified_at: string | null
      expires_at: string | null
      created_at: string
    }

    export interface VerifyStartResponse {
      domain: string
      token: string
      meta_tag: string
      already_verified?: boolean
      expires_in_days?: number
    }

    export interface VerifyConfirmResponse {
      verified: boolean
      domain: string
      expires_at?: string
      failure_reason?: string
    }

    export interface VerifyCheckResponse {
      found: boolean
      domain: string
      message: string
    }
    ```

    **API functions** (`frontend/lib/api.ts`) — add these functions using the existing `BACKEND_URL` pattern from the file:

    1. `verifyStart(domain: string, authToken: string): Promise<VerifyStartResponse>` — POST to `/api/v1/domains/verify-start` with `Authorization: Bearer ${authToken}` and `Content-Type: application/json`. Body: `{ domain }`. Throw on non-OK with error detail from response.

    2. `verifyConfirm(domain: string, authToken: string): Promise<VerifyConfirmResponse>` — POST to `/api/v1/domains/verify-confirm` with same auth pattern. Body: `{ domain }`.

    3. `verifyCheck(domain: string, authToken: string): Promise<VerifyCheckResponse>` — POST to `/api/v1/domains/verify-check` with same auth pattern. Body: `{ domain }`.

    4. `listDomains(authToken: string): Promise<VerifiedDomain[]>` — GET `/api/v1/domains` with auth header. Return empty array on error (graceful degradation, same as existing patterns).

    Use the existing `NEXT_PUBLIC_BACKEND_URL` env var for client-side API calls. The dashboard server-side fetch uses `BACKEND_URL` (server-only env var).

    **Route protection** (`frontend/proxy.ts`):
    Update `isProtectedRoute` matcher from:
    ```ts
    const isProtectedRoute = createRouteMatcher(['/dashboard(.*)'])
    ```
    to:
    ```ts

    ```
    This ensures unauthenticated users are redirected to sign-in when visiting /verify-domain.

  <action>
    **DomainBadge** (`frontend/components/domain-badge.tsx`):
    A compact pill badge showing domain verification status. NOT a `'use client'` component — it's a presentational server component.

    Props: `{ status: 'pending' | 'verified', expiresAt: string | null }`

    Logic:
    - If `status === 'verified'` and `expiresAt`:
      - Parse `expiresAt` as Date, compute days remaining: `Math.ceil((expires - now) / (1000 * 60 * 60 * 24))`
      - If `days <= 0`: Red "Expired" badge — `bg-danger-bg text-danger-text border border-danger-border` (uses existing design tokens from globals.css)
      - If `days <= 7`: Yellow "Expires in {N}d" badge — `bg-caution-bg text-caution-text border border-caution-border`
      - If `days > 7`: Green "Verified" badge — `bg-success-bg text-success-text border border-success-border`
    - If `status === 'pending'`: Blue "Pending" badge — `bg-info-bg text-info-text border border-info-border`
    - Badge shape: `inline-flex items-center px-2 py-0.5 rounded-full text-xs font-medium`
    - Use Lucide icons: `CheckCircle2` for verified, `Clock` for pending, `AlertTriangle` for expiring, `XCircle` for expired

    **MetaTagSnippet** (`frontend/components/meta-tag-snippet.tsx`):

    - Button: `absolute top-3 right-3 p-1.5 rounded hover:bg-gray-700 transition-colors`
    - Aria label: "Copy meta tag"

    **Verify Domain Wizard** (`frontend/app/verify-domain/page.tsx`):
    A `'use client'` component — needs Clerk auth hooks, useState for step progression, and API calls.

    State machine: `type WizardStep = 'input' | 'snippet' | 'verifying' | 'success' | 'failed'`

    State variables:
    - `step: WizardStep` (initially 'input')
    - `domain: string` (user input)
    - `verifyData: VerifyStartResponse | null` (from verify-start)
    - `failureReason: string | null` (from verify-confirm failure)
    - `error: string | null` (validation/API errors)
    - `loading: boolean` (for async operations)
    - `testResult: { found: boolean; message: string } | null` (from verify-check)

    **Step 'input':**
    - Text input for domain with placeholder "myapp.vercel.app"
    - **Frontend blocklist check**: Before calling API, check if the input (after basic normalization — trim, strip scheme/path, strip www) exactly matches a blocked root TLD. If so, show inline error: "'{domain}' is a shared hosting platform. Enter your app's subdomain instead (e.g., myapp.{domain})."
    - "Start Verification" button calls `verifyStart(domain, token)` via `useAuth().getToken()`
    - On success:
      - If `already_verified === true`: show message "This domain is already verified. Expires in {expires_in_days} days." with a "Return to Dashboard" link. Do NOT advance to snippet step.
      - Otherwise: set `verifyData`, advance to 'snippet' step
    - On error: display error message inline

    **Step 'snippet':**
    - Display instructions: "Add this meta tag to the `<head>` section of your site at https://{domain}/"
    - Render `<MetaTagSnippet metaTag={verifyData.meta_tag} />`
    - Two buttons:
      1. "Test my tag" (secondary) — calls `verifyCheck(domain, token)`. Shows result: green check "Tag found!" or red warning with specific diagnosis message. Does NOT advance step.
      2. "Verify now" (primary) — sets step to 'verifying', calls `verifyConfirm(domain, token)`

    **Step 'verifying':**
    - Loading spinner with "Verifying your domain..."
    - On `verifyConfirm` success (`verified: true`): advance to 'success'
    - On `verifyConfirm` failure (`verified: false`): set `failureReason`, advance to 'failed'

    **Step 'success':**
    - Green check icon with "Domain verified!" heading
    - Show domain name and expiry date: "Expires {expires_at formatted}"
    - "Go to Dashboard" button using `router.push('/dashboard')`
    - Per user decision: one domain at a time, return to dashboard after success

    **Step 'failed':**
    - Red X icon with "Verification failed" heading
    - Show `failureReason` with specific diagnosis (from verify-confirm response)
    - "Try again" button resets to 'snippet' step (user can re-check tag placement)

    **Page layout:**
    - Centered card layout with max-w-lg, padding, and card styling consistent with existing pages
    - Page title: "Verify Domain Ownership"
    - Use semantic tokens for backgrounds and text
