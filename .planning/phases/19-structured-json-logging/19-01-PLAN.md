**src/main.rs changes — replace the entire logging initialization section:**

1. **Remove RUST_LOG from required env vars.** Update `validate_required_env_vars` call to remove "RUST_LOG" from the list. RUST_LOG is now optional — sensible defaults handle everything.

2. **Create a `build_env_filter()` function** that builds an EnvFilter with sensible defaults:
   - Use `cfg!(debug_assertions)` to detect debug vs release builds
   - Debug default: `"debug,hyper=info,sqlx=info,tower=info,tower_http=info,reqwest=info,h2=info"`
   - Release default: `"info,hyper=warn,sqlx=warn,tower=warn,tower_http=warn,reqwest=warn,h2=warn"`
   - Use `EnvFilter::builder().with_default_directive(...)` — NO, this doesn't work as a multi-directive default. Instead, use `EnvFilter::try_new(default_directives).unwrap()` as the fallback, and check if RUST_LOG is set: if set, use `EnvFilter::try_from_default_env().unwrap_or_else(|_| ...)` which reads RUST_LOG and overrides completely; if not set, use the default filter.
   - Correct approach: `EnvFilter::builder().with_env_var("RUST_LOG").with_default_directive(tracing_subscriber::filter::LevelFilter::INFO.into()).parse_lossy(default_directives)` — Actually the cleanest pattern per research is:
     ```rust
     fn build_env_filter() -> EnvFilter {
         // RUST_LOG overrides everything when set
         if std::env::var("RUST_LOG").is_ok() {
             return EnvFilter::from_default_env();
         }
         // Sensible defaults based on build profile
         let defaults = if cfg!(debug_assertions) {
             "debug,hyper=info,sqlx=info,tower=info,tower_http=info,reqwest=info,h2=info"
         } else {
             "info,hyper=warn,sqlx=warn,tower=warn,tower_http=warn,reqwest=warn,h2=warn"
         };
         EnvFilter::new(defaults)
     }
     ```

3. **Create `init_logging()` function** that:
   - Reads LOG_FORMAT env var: `std::env::var("LOG_FORMAT").unwrap_or_default()`
   - If LOG_FORMAT == "json": initialize with `.json()` formatter — `tracing_subscriber::fmt().json().with_env_filter(build_env_filter()).with_target(true).with_thread_ids(false).with_thread_names(false).init()`
   - Otherwise (text mode): initialize with `.with_ansi(false)` — `tracing_subscriber::fmt().with_ansi(false).with_env_filter(build_env_filter()).with_target(true).init()`
   - Both modes include timestamp, level, target automatically (tracing-subscriber defaults).

4. **Install panic hook** after logging init:
   ```rust
   let prev_hook = std::panic::take_hook();
   std::panic::set_hook(Box::new(move |panic_info| {
       tracing::error!(
           panic.message = %panic_info,
           panic.location = panic_info.location().map(|l| l.to_string()).unwrap_or_default(),
           "A panic occurred"
       );
       prev_hook(panic_info);
   }));
   ```
   Note: Use this manual approach instead of `tracing-panic` if `tracing_panic::panic_hook` does not chain to previous hook or does not include backtrace. Check: `tracing-panic` 0.1 provides `panic_hook` that logs with tracing but we need backtrace. The simpler manual approach above gives us control. Use `tracing-panic` if it provides what we need — it logs panic info as a tracing event at ERROR level. Per the research, `use tracing_panic::panic_hook; std::panic::set_hook(Box::new(panic_hook));` is the pattern. This replaces the default hook. For backtrace, set RUST_BACKTRACE=1 — the panic info itself includes location. Use the tracing-panic crate as recommended by research.

   Final approach: Use tracing-panic:
   ```rust
   use tracing_panic::panic_hook;
   std::panic::set_hook(Box::new(panic_hook));
   ```

5. **Add startup banner** after logging init and DB connection, as a single info event:

   ```rust
       log_filter = %filter_description,
       port = port,
       db_connected = true,

   );
   ```
   Where `log_format` is "json" or "text", and `filter_description` is "RUST_LOG" if RUST_LOG is set, or "defaults (debug)" / "defaults (release)" otherwise. Move the existing `tracing::info!("TrustEdge Audit API listening on {}", addr)` to use structured fields too.
