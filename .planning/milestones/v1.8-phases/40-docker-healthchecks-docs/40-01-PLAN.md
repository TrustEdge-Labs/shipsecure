  <name>Task 1: Add Docker healthchecks to production containers</name>
  <files>frontend/app/api/health/route.ts, docker-compose.prod.yml</files>
  <action>
  **Step 1: Create frontend health endpoint.**

  Create `frontend/app/api/health/route.ts` — a Next.js App Router API route that returns a simple 200 response. This gives the Docker healthcheck a lightweight endpoint to probe without rendering a full page.

  ```typescript
  import { NextResponse } from "next/server";

  export async function GET() {
    return NextResponse.json({ status: "ok" });
  }
  ```

  **Step 2: Add healthcheck to backend service in docker-compose.prod.yml.**

  Add a `healthcheck:` block to the `backend` service using the locked decision parameters:
  - Command: `curl -f http://localhost:3000/health/ready || exit 1`
  - Interval: 30s
  - Timeout: 30s
  - Retries: 3
  - Start period: 60s (backend needs time to initialize DB connections and scan infrastructure)

  The `/health/ready` readiness endpoint already exists in `src/api/health.rs` — it validates DB connectivity, scan capacity, and cache. It returns 503 during graceful shutdown, which makes Docker mark the container unhealthy before it stops.

  `curl` is already installed in the backend runtime image (`debian:bookworm-slim`).

  **Step 3: Add healthcheck to frontend service in docker-compose.prod.yml.**

  Add a `healthcheck:` block to the `frontend` service:
  - Command: `wget --no-verbose --tries=1 --spider http://localhost:3001/api/health || exit 1`
  - Interval: 30s
  - Timeout: 10s
  - Retries: 3
  - Start period: 30s (Next.js starts faster than the backend)

  `wget` is available in `node:20-alpine` (the frontend runtime image) — no extra packages needed. Using `--spider` makes a HEAD-like request without downloading the body. Using `--no-verbose` keeps healthcheck logs clean.

  **Step 4: Upgrade depends_on condition.**

  Change the frontend's `depends_on` from:
  ```yaml
  depends_on:
    backend:
      condition: service_started
  ```
  to:
  ```yaml
  depends_on:
    backend:
      condition: service_healthy
  ```

  This ensures the frontend container only starts after the backend healthcheck passes, meaning the backend is genuinely ready to serve requests (DB connected, scan capacity available).
