Create `src/scanners/js_secrets.rs` implementing SCAN-05 (JavaScript secret scanning).

**Design:**
- Public async function: `pub async fn scan_js_secrets(url: &str) -> Result<Vec<Finding>, ScannerError>`
- Reuse `ScannerError` pattern from `security_headers.rs`

**Step 1: Discover JavaScript bundle URLs**
1. Fetch the target URL's HTML page
2. Parse response body for `<script src="...">` tags using simple regex (not a full HTML parser â€” keep dependencies minimal): `<script[^>]+src=["']([^"']+\.js[^"']*)["']`
3. Resolve relative URLs to absolute URLs using the base URL
4. Also check common bundle paths: `/main.js`, `/app.js`, `/bundle.js`, `/static/js/main.*.js`, `/_next/static/chunks/*.js`
5. Deduplicate discovered URLs
6. Limit to 20 JS files max (prevent abuse on sites with hundreds of scripts)

**Step 2: Fetch and scan each JavaScript file**
For each JS URL (concurrent with `futures::future::join_all`, 10s timeout each):
1. Fetch the JS content via HTTP GET
2. Limit to first 2MB of content per file (prevent memory issues)
3. Run pattern matching against the content

**Secret patterns to detect (with confidence levels):**

HIGH confidence (format-validated):
- AWS Access Key: `AKIA[0-9A-Z]{16}` -- severity: Critical
- Stripe Live Secret Key: `sk_live_[a-zA-Z0-9]{24,}` -- severity: Critical
- Stripe Live Publishable Key: `pk_live_[a-zA-Z0-9]{24,}` -- severity: Medium (publishable keys are meant to be public, but flag for awareness)
- GitHub Token: `ghp_[a-zA-Z0-9]{36}` or `gho_[a-zA-Z0-9]{36}` -- severity: Critical
- Slack Token: `xox[bprs]-[a-zA-Z0-9-]+` -- severity: High
- Twilio API Key: `SK[a-f0-9]{32}` -- severity: High

MEDIUM confidence (pattern + context):
- Supabase Anon Key: `eyJ[a-zA-Z0-9_-]+\.eyJ[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+` near `supabase` -- severity: Medium (anon keys are semi-public but should be reviewed)
- Firebase Config: `apiKey\s*[:=]\s*["'][A-Za-z0-9_-]{39}["']` near `firebase` -- severity: Medium
- Generic API Key: `(?:api[_-]?key|apikey)\s*[:=]\s*["'][a-zA-Z0-9_-]{20,}["']` -- severity: Medium

LOW confidence (entropy-based, needs filtering):
- High-entropy strings (>4.5 bits/char, >20 chars) near keywords like `secret`, `password`, `token`, `key`, `credential` -- severity: Low

**False positive filtering:**
- Skip strings matching common test patterns: contains "test", "example", "placeholder", "YOUR_KEY_HERE", "xxx", "000", "123", "CHANGE_ME", "todo"
- Skip strings that are all same character repeated
- Skip Stripe test keys: `sk_test_`, `pk_test_`
- Skip known public/example values: Firebase demo project keys, Supabase example keys
- For entropy-based matches, require the keyword to be within 50 characters of the high-entropy string

**Finding generation:**
- `scanner_name`: "js_secrets"
- `title`: e.g., "Hardcoded AWS Access Key in JavaScript"
- `description`: "A hardcoded {key_type} was found in {js_file_url}. Hardcoded secrets in client-side JavaScript are visible to anyone who views your site."
- `remediation`: "Remove the secret from your JavaScript code. Use environment variables on the server side. For {key_type}: {specific_advice}" (e.g., for AWS: "Rotate this key immediately in the AWS IAM console and use AWS Cognito or a backend proxy for client-side access.")
- `raw_evidence`: First 100 chars around the match (redact middle of the key: show first 8 chars + "..." + last 4 chars)

Add `regex` crate to Cargo.toml: `regex = "1"` and `futures = "0.3"` (for join_all).

Create `src/scanners/container.rs` implementing INFRA-05 (containerized scanner execution with security hardening).

**Design:**
- Struct `ContainerRunner` with configurable timeout and resource limits
- Method: `pub async fn run_nuclei(target: &str) -> Result<String, ScannerError>`
- Method: `pub async fn run_testssl(target: &str) -> Result<String, ScannerError>`
- Helper: `async fn run_docker_container(image: &str, args: &[&str], timeout: Duration, memory_limit: &str) -> Result<String, ScannerError>`

**Docker security hardening (MANDATORY for all containers):**
All docker run commands MUST include:
- `--rm` (auto-remove after exit)
- `--read-only` (read-only filesystem)
- `--cap-drop all` (drop all Linux capabilities)
- `--user 1000:1000` (non-root user)
- `--memory {limit}` (memory limit: 100M for testssl, 512M for nuclei)
- `--pids-limit 1000` (process limit)
- `--cpu-shares 512` (CPU limit)
- `--network=host` is NOT used (use default bridge networking)
- `--no-new-privileges` (prevent privilege escalation)

**Nuclei execution:**
```
docker run --rm --read-only --cap-drop all --user 1000:1000 \
  --memory 512M --pids-limit 1000 --cpu-shares 512 --no-new-privileges \
  projectdiscovery/nuclei:latest \
  -u {target} -jsonl -silent -severity medium,high,critical \
  -tags exposure,misconfig,cve
```
- Parse JSONL output (one JSON object per line)
- Map nuclei severity to Finding severity
- Use nuclei's `info.name` as finding title, `info.description` as description
- Timeout: 120 seconds (nuclei scans can take longer)

**testssl.sh execution:**
```
docker run --rm --read-only --cap-drop all --user 1000:1000 \
  --memory 100M --pids-limit 1000 --cpu-shares 512 --no-new-privileges \
  drwetter/testssl.sh:latest \
  --jsonfile-pretty /dev/stdout --quiet {target}
```
- Parse JSON output
- Map testssl severity ("CRITICAL", "HIGH", "MEDIUM", "LOW", "INFO", "OK") to Finding severity
- Use testssl `id` as finding identifier
- Timeout: 180 seconds (TLS scans are thorough)

**Docker availability check:**
- Add function `pub async fn is_docker_available() -> bool` that runs `docker info` and checks exit code
- If Docker is not available, scanners should return an empty Vec<Finding> with a warning log, NOT an error (allows development without Docker)

**Error handling:**
- Container timeout: Kill container via timeout, return ScannerError with descriptive message
- Container exit code != 0: Parse stderr for error message
- Docker not installed: Log warning, return empty findings
- Image not pulled: Let Docker auto-pull (first run will be slow)

Use `tokio::process::Command` for async Docker execution. Set environment variable `DOCKER_AVAILABLE` check at startup to avoid repeated `docker info` calls.

Update `src/scanners/mod.rs` to include: `pub mod container;`
