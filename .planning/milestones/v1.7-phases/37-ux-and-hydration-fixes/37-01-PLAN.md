  <name>Task 1: Investigate and fix React hydration mismatch</name>
  <files>frontend/app/layout.tsx</files>
  <action>
React hydration mismatches in Next.js App Router most commonly come from:
1. Browser extensions injecting attributes onto `<html>` or `<body>` (color-scheme, data-*, class)
2. Clerk injecting its own attributes or theme tokens into the DOM server-side vs client-side
3. Dark mode class/attribute set differently on server vs client

The current layout.tsx already has `suppressHydrationWarning` on `<html lang="en">`. This is the correct placement per React docs — it suppresses warnings for the html element's own attributes (e.g. browser extensions adding data-* attrs).

However, if warnings still appear, the `<body>` element may also need `suppressHydrationWarning` if extensions or Clerk modify it. Additionally, the ClerkProvider `appearance` prop, if used, must not produce server/client differences.

Steps:
1. Read the current `frontend/app/layout.tsx` carefully.
2. Add `suppressHydrationWarning` to the `<body>` element as well — this is a low-risk, high-impact fix for extension-injected attributes (class, style, data-gr-c-s-loaded, etc.).
3. Do NOT add a ClerkProvider `appearance` prop unless there is clear evidence it is needed — the issue is almost certainly browser extension attribute injection on `<body>`, not Clerk.
4. The `<html>` already has `suppressHydrationWarning` — leave it as-is.
5. Result: both `<html>` and `<body>` carry `suppressHydrationWarning`, which is the canonical Next.js pattern for suppressing browser-extension-induced mismatch warnings.

Do not add `suppressHydrationWarning` to any other elements — it silences ALL hydration warnings for that subtree, which would hide real bugs.
