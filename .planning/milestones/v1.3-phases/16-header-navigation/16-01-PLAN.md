<task type="auto">
  <name>Task 1: Create header component with responsive logo and CTA</name>
  <files>frontend/components/header.tsx</files>
  <action>
Create a new Server Component at `frontend/components/header.tsx` (NO "use client" directive). The header must:

1. **Structure**: Use `<header>` element with `<nav>` inside for semantic HTML. Apply classes: `sticky top-0 z-50 bg-surface-primary border-b border-border-subtle`. These semantic tokens match the footer pattern established in Phase 13.

2. **Container**: Inner `<div>` with `container mx-auto px-4 h-[var(--header-height)] flex items-center justify-between`. The `h-[var(--header-height)]` consumes the 64px token from Phase 15.

3. **Logo link (left side)**: Wrap in `<Link href="/">` with `className="flex items-center"`. Inside, render TWO variants using Tailwind visibility:
   - Desktop (>=640px): `<div className="hidden sm:block">` containing `<Logo size="small" />` — this is 96x64 from logo.tsx sizeMap, which fits the 64px header height.
   - Mobile (<640px): `<div className="sm:hidden">` containing `<Image src="/logo.png" alt="ShipSecure" width={40} height={40} priority className="object-contain" />` — a 40x40 icon-sized rendering of the logo. Use next/image directly here because the Logo component's smallest size (96x64) is too wide for mobile header. The `object-contain` ensures the shield icon portion of the logo renders clearly at this small size.

   Note: Both use `priority` prop (the Logo component already sets `priority` only for "large" size, so the "small" variant needs it explicitly). Add `priority` prop to the Logo component usage: `<Logo size="small" className="..." />` — but Logo component doesn't pass priority for small. Instead, render the desktop variant the same way as mobile: directly with `<Image>`. This keeps both consistent:
   - Desktop: `<Image src="/logo.png" alt="ShipSecure" width={160} height={48} priority className="object-contain" />` — wordmark visible at desktop width
   - Mobile: `<Image src="/logo.png" alt="ShipSecure" width={40} height={40} priority className="object-contain" />` — icon mark only at mobile width

   Use next/image directly for BOTH rather than the Logo component, because:
   - We need `priority` on both (Logo only sets it for "large")
   - We need custom dimensions that don't match Logo's predefined sizes
   - Simpler, more explicit, no abstraction leakage

4. **CTA button (right side)**: `<Link href="/#scan-form">` with classes: `px-4 py-2 bg-brand-primary hover:bg-brand-primary-hover text-text-inverse font-semibold rounded-lg transition text-sm sm:text-base`. Text content: "Scan Now". The `/#scan-form` hash link scrolls to the scan form on homepage and navigates-then-scrolls from other pages.

5. **Accessibility**: Add `role="banner"` is implicit on `<header>` (don't add redundant role). The `<nav>` should have `aria-label="Main navigation"`. Ensure all interactive elements (logo link, CTA link) are naturally focusable via `<Link>` which renders `<a>` tags.

6. **Export**: Named export `export function Header()` matching the footer pattern.

Do NOT add:
- "use client" directive (no interactivity needed)
- usePathname or active link styling (not in requirements)
- Mobile hamburger menu (not in requirements)
- Additional nav links beyond logo and CTA (Privacy/Terms stay in footer only per research recommendation)

**Edit 2: frontend/app/globals.css**
Add a `:root` rule AFTER the body styles block (after line 317) for scroll-padding-top:

```css
/* ===================================================================
   Accessibility: Sticky Header Focus Protection (WCAG 2.2 SC 2.4.11)
   =================================================================== */

:root {
  scroll-padding-top: var(--header-height);
}
```

This ensures keyboard-focused elements are never obscured behind the sticky header. The var(--header-height) reference uses the 64px token defined in the @theme inline block. Place this OUTSIDE @theme inline because scroll-padding-top is a CSS property, not a theme token.
